# OSINT 기반 군사 이벤트 탐지 및 위성 촬영 후보 선정 대시보드

## 1. 프로젝트 개요

본 프로젝트는 OSINT 기반 글로벌 이벤트 데이터, 위성 궤도 정보, 기상 조건을 결합해 군사 이벤트 발생 지역의 위성 촬영 후보를 빠르게 선별하는 의사결정 지원 대시보드입니다.

위성 영상 분석 업무에서는 사건 발생 후 촬영 가능한 시간이 제한적입니다. 군사 시설, 장비 이동, 폭격 흔적과 같은 물리적 증거는 시간이 지나면 은폐되거나 훼손될 수 있기 때문에, 이벤트 발생 지역을 빠르게 포착하고 촬영 가능 조건을 판단하는 과정이 중요합니다.

기존에는 분석관이 다음 작업을 개별적으로 수행해야 했습니다.

* OSINT 뉴스 및 이벤트 데이터 확인
* 군사 이벤트 발생 지역 선별
* 위성 통과 가능 시간 확인
* 날씨와 주야간 조건 확인
* 촬영 후보 지역 우선순위 판단

이 프로젝트는 위 과정을 하나의 파이프라인으로 통합해, 분석관이 촬영 후보 지역과 근거 정보를 한 화면에서 확인할 수 있도록 구성했습니다.

## 2. 문제 정의

프로젝트의 핵심 문제는 다음과 같습니다.

> 군사 이벤트가 발생했을 가능성이 높은 지역을 빠르게 탐지하고, 실제 위성 촬영이 가능한 후보로 연결할 수 있는가?

이를 위해 단순히 이벤트 수를 많이 보여주는 것이 아니라, 분석관이 다음 판단을 내릴 수 있도록 다음 정보를 함께 제공하는 것을 목표로 했습니다.

* 어디에서 군사 이벤트 신호가 증가했는가
* 그 신호가 평소 대비 얼마나 이례적인가
* 실제 기사 본문에서도 군사 이벤트로 확인되는가
* 해당 지역을 촬영할 수 있는 위성이 언제 지나가는가
* 촬영 시점의 기상 조건은 적절한가
* 최종적으로 어떤 후보를 우선 검토해야 하는가

## 6. 데이터 전처리

전처리 단계에서는 다음 작업을 수행했습니다.

1. 모니터링 대상 국가 필터링
2. 물리적 충돌 관련 CAMEO 코드 필터링
3. 도시 단위 이벤트만 추출
4. 중복 이벤트 제거
5. 지명 표준화
6. 좌표 오류 보정
7. 블랙리스트 기반 노이즈 제거

### 6.1 CAMEO 코드 필터링

| RootCode | 의미              |
| -------- | --------------- |

이를 통해 외교 발언, 단순 시위, 일반 정치 이벤트보다 실제 촬영 필요성이 높은 이벤트에 집중했습니다.

## 7. Conflict Index 설계

도시별 군사 이벤트 신호를 정량화하기 위해 `Conflict Index`를 정의했습니다.

초기에는 `NumMentions`, `NumSources`, `AvgTone`, `GoldsteinScale` 등 여러 변수를 검토했지만, 도시-일자 단위에서 가장 안정적인 신호는 독립 보도 출처 수를 의미하는 `NumSources`라고 판단했습니다.

최종적으로는 다음과 같은 기준을 적용했습니다.

```text
Conflict Index = Σ log(1 + NumSources)
```

이 방식은 단일 기사보다 여러 독립 출처에서 반복적으로 보도된 도시를 더 강한 신호로 반영합니다.

평균값이 아니라 합계를 사용한 이유는 이벤트 규모 정보를 유지하기 위해서입니다. 같은 평균값을 가진 도시라도 이벤트가 1건인 도시와 100건인 도시는 촬영 우선순위가 달라야 하기 때문입니다.

## 8. Kalman Filter 기반 이상탐지

도시별 Conflict Index는 지역마다 baseline이 크게 다릅니다. 예를 들어 Tehran처럼 평소 보도량이 많은 도시는 높은 값이 정상일 수 있고, 작은 국경 도시에서는 작은 증가도 중요한 신호일 수 있습니다.

따라서 전체 도시를 같은 절대 기준으로 비교하기보다, 도시별 평소 baseline 대비 급증 여부를 판단해야 했습니다.

이를 위해 1D Kalman Filter를 적용했습니다.

### 8.1 Kalman Filter 적용 이유

ARIMA는 도시별 시계열에 대해 개별 파라미터 추정이 필요하고, 충분한 관측치가 없는 도시에는 적용하기 어렵습니다. 반면 Kalman Filter는 비교적 짧고 불규칙한 도시별 시계열에도 적용 가능하며, 도시 수가 많아도 선형적으로 확장할 수 있습니다.

### 8.2 이상점수

Kalman Filter에서는 관측값과 예측 baseline의 차이인 innovation을 계산하고, 이를 표준화해 도시 간 비교 가능한 anomaly score로 사용했습니다.

```text
innovation = observed_conflict_index - predicted_baseline
standardized_innovation = innovation / sqrt(S)
```

이 표준화된 innovation score를 기반으로 관심, 주의, 심각 리스크 레벨을 부여했습니다.

## 9. Ground Truth 및 평가 설계

이상탐지 성능을 평가하기 위해 공개 전황 보고서와 사건 기록을 참고해 도시-일자 단위 검증 라벨을 수동으로 구축했습니다.

평가 단위는 다음과 같습니다.

```text
(date, standard_name)
```

즉, 특정 날짜에 특정 도시에서 실제 군사 이벤트가 발생했는지를 Positive 기준으로 설정했습니다.

### 9.1 평가 기준

| 항목             | 설명                                  |
| -------------- | ----------------------------------- |
| 평가 단위          | 도시-일자 `(date, standard_name)`       |
| Positive       | 공개 전황 보고서 및 사건 기록에서 확인된 이벤트 도시      |
| Negative       | 동일 기간 내 Ground Truth에 포함되지 않은 도시-일자 |
| 주요 지표          | ROC-AUC, Recall                     |
| Recall을 중시한 이유 | 촬영 후보 선정 문제에서는 실제 이벤트 누락 비용이 크기 때문  |

### 9.2 Kalman Filter vs ARIMA

동일한 Ground Truth 기준에서 Kalman Filter와 ARIMA의 이상탐지 성능을 비교했습니다.

| 모델            | ROC-AUC | 적용 가능 도시 수 |
| ------------- | ------: | ---------: |
| Kalman Filter |   0.762 |      1,115 |
| ARIMA         |   0.620 |        286 |

Kalman Filter는 ARIMA보다 ROC-AUC가 높았고, 적용 가능한 도시 수도 더 많았습니다. 특히 ROC 초반 구간에서 상대적으로 높은 TPR을 보여, 낮은 오탐률 조건에서도 실제 이벤트가 발생한 도시-일자 후보를 더 많이 탐지하는 경향을 확인했습니다.

## 10. LLM 기반 기사 검증

이상탐지 결과만으로는 도시명 오류, 과거 사건 재언급, 단순 정치 기사 등이 후보에 포함될 수 있습니다. 따라서 상위 anomalous city에 대해 실제 기사 본문을 읽고 군사 이벤트 여부를 검증하는 LLM 단계를 추가했습니다.

### 10.1 검증 방식

1. 이상점수가 높은 도시 후보 선정
2. 해당 도시와 관련된 기사 URL 수집
3. 기사 본문 scraping
4. 도시명 주변 문맥 추출
5. LLM이 기사 내용과 날짜 조건을 기준으로 라벨 분류

### 10.2 LLM 라벨

| 라벨              | 의미                           |
| --------------- | ---------------------------- |
| `SUCCESS`       | 직접 공격, 폭격, 교전 등 실제 군사 이벤트 확인 |
| `AMBIGUOUS`     | 간접 군사 신호 또는 날짜 불확실           |
| `DATE_MISMATCH` | 군사 이벤트는 있으나 평가 기간 밖 사건       |
| `DROPPED`       | 도시명이 단순 언급되었거나 군사 이벤트 아님     |
| `NO_MENTION`    | 기사 본문에서 도시가 의미 있게 언급되지 않음    |

`SUCCESS`와 `AMBIGUOUS`는 스케줄링 후보로 넘기고, `DATE_MISMATCH`, `DROPPED`, `NO_MENTION`은 필터링했습니다.

## 11. 위성 촬영 후보 선정

LLM 검증을 통과한 도시 후보에 대해 위성 촬영 가능성을 계산했습니다.

### 11.1 고려 조건

| 조건              | 설명                  |
| --------------- | ------------------- |
| 위성 통과 시간        | TLE 기반 궤도 계산        |
| off-nadir angle | 촬영 가능 각도 제한         |
| 구름량             | EO 위성 촬영 가능성 판단     |
| 주야 조건           | 광학 촬영 가능 여부 판단      |
| 센서 유형           | EO / SAR 구분         |
| 우선순위            | 후보 위성별 실무 활용 가능성 반영 |

### 11.2 촬영 가능성 판단

EO 위성은 구름량과 주간 조건의 영향을 받기 때문에 촬영 가능 여부를 별도로 계산했습니다. SAR 위성은 구름과 주야간 제약이 상대적으로 작기 때문에 별도 후보군으로 표시했습니다.

최종적으로 도시별 촬영 후보를 생성하고, 대시보드에서 분석관이 우선순위를 검토할 수 있도록 구성했습니다.

## 17. 내가 집중한 부분

이 프로젝트에서 가장 중요하게 본 지점은 분석 결과를 단순히 보여주는 것이 아니라, 분석관이 다음 판단을 내릴 수 있는 형태로 구조화하는 것이었습니다.

따라서 모델 성능만이 아니라 다음 흐름을 함께 고려했습니다.

* 어떤 이벤트를 촬영 후보로 볼 것인가
* 어떤 도시의 신호가 평소 대비 이례적인가
* LLM 검증 결과를 어떻게 후보 필터링에 활용할 것인가
* 위성 통과 조건과 기상 조건을 어떻게 함께 보여줄 것인가
* 분석관이 최종 판단을 내리기 위해 어떤 정보를 한 화면에서 봐야 하는가

## 18. 한계

이 프로젝트는 실제 위성 tasking 시스템이 아니라, 기업 연계 과제 상황을 기반으로 만든 촬영 후보 선정 MVP입니다.

다음과 같은 한계가 있습니다.

* GDELT에 보도되지 않은 사건은 탐지할 수 없음
* 영어권 뉴스 중심 보도 편향이 존재함
* 소규모 도시나 보도량이 적은 지역의 Recall이 낮음
* LLM 검증 결과는 기사 접근성, 기사 품질, 날짜 표현에 영향을 받음
* 실제 촬영 명령 발행, 비용 최적화, 위성 운영 제약까지는 포함하지 않음
* Ground Truth는 공개 전황 보고서와 사건 기록을 기반으로 수동 구축했기 때문에 완전한 정답셋은 아님

## 19. 후속 개선 방향

프로젝트를 확장한다면 다음 작업을 추가할 수 있습니다.

1. Ground Truth 고도화

   * ISW, Wikipedia 외 추가 공개 전황 데이터 반영
   * 사건 유형별 라벨 세분화
   * 도시-일자-이벤트 유형 단위 평가로 확장

2. 이상탐지 모델 개선

   * Prophet, Bayesian Online Change Point Detection 등 추가 baseline 비교
   * 도시 규모별 threshold 최적화
   * false positive / false negative 사례 분석 강화

3. LLM 검증 개선

   * 기사별 evidence span 저장
   * LLM 판단 근거 구조화
   * LLM label consistency 평가

4. 촬영 후보 최적화

   * 위성별 비용, 해상도, revisit time 반영
   * 촬영 우선순위 scoring 고도화
   * 다중 도시 촬영 경로 최적화

5. 운영 관점 보완

   * 일별 자동 실행 pipeline
   * 데이터 업데이트 모니터링
   * 분석관 feedback 기반 후보 품질 개선

## 20. 정리

이 프로젝트는 OSINT 이벤트 데이터를 시각화하는 데서 끝나는 프로젝트가 아닙니다.

GDELT 기반 이벤트 탐지, 도시별 이상탐지, LLM 기사 검증, 위성·기상 조건 계산을 결합해 분석관이 더 빠르고 근거 있게 촬영 후보를 판단할 수 있도록 설계한 데이터 제품 MVP입니다.

핵심은 데이터를 많이 보여주는 것이 아니라, 사용자가 다음 판단을 내릴 수 있는 구조로 데이터를 정리하는 것이었습니다.
